Platform Overview — What is ALTRIX?
Mission, purpose, and the 3-tool ecosystem

ALTRIX is a next-generation AI research platform that packages three specialised AI engines into one cohesive intelligence suite. It is built for researchers, academics, and writers who need to humanize AI text, analyze research papers, and search across live academic databases — all without leaving a single platform.

The tagline "Intelligence, Amplified" reflects the dual mission: ALTRIX is not just an AI tool, it is an amplifier of human cognitive work. Each of the three tools tackles a distinct stage of the research and writing workflow.

ALTRIX runs entirely on Next.js 14 with the App Router. There is no separate backend server for the main application — all API routes run as serverless edge functions within the Next.js process itself. The only separate backend is the Python FastAPI server (port 8000) used for SENTIC's marker injection and ALAMIX's document text extraction — processes that require native Python libraries (python-docx, PyMuPDF, transformers).

Two Runtime Boundary

All Groq API calls happen server-side inside Next.js API routes — the API key is never exposed to the browser. The Python FastAPI is called by the Next.js API routes as a proxied internal request (http://localhost:8000), not directly by the frontend.

Route Group Behaviour

The (dashboard) folder name with parentheses is a Next.js route group — it groups related pages under a shared layout without adding a URL segment. So /sentic, /alamix, and /deoza all get the NavigationRail automatically, while the root / landing page remains clean and standalone.

The Three AI Tools — High-Level Summary
What each tool does, what powers it, and its output

SENTIC — Text Humanization Engine

SENTIC transforms AI-generated text into authentic-sounding human writing by attacking AI detection at three distinct layers simultaneously.

Layer What it does How
Pre-processing Strip AI-isms from input Regex ban-list of words like "comprehensive", "delve", "leverage"
Stage 1 Linguistic Entropy rewrite Groq LLaMA 70B — Perplexity + Burstiness master prompt. Chunks text if >300 words
Stage 2 Pragmatic marker injection Python FastAPI + custom algorithm — injects human discourse signals ("Frankly", "In practice")
Client-side AI Score Estimator Formula: sentence length variance + banned word hits + punctuation richness
Post-processing Text cleanup settings User-controlled: remove Unicode, dashes, quotes, whitespace

Output panels: Humanized Text · AI Score Badge · Pipeline Log · Entropy Parameters · Explainability Log

ALAMIX — Research Intelligence Engine

ALAMIX uploads a PDF or DOCX, extracts raw text via Python, then sends it to Groq for structured academic analysis at temperature=0 (fully deterministic).

Stage What it does How
Frontend File upload, simulated progress (4 steps) React state, FormData, drag-drop
Stage 1 Text extraction from PDF/DOCX Python FastAPI /extract-text endpoint (PyMuPDF / python-docx)
Stage 2 Academic intelligence extraction Groq at temperature=0, JSON mode — 16-field structured output
Output UI Intelligence Report 12-column grid, 15 expandable AcademicCards, sticky sidebar
Chatbot Paper Q&A DEOZA chat endpoint (/api/deoza/chat) with paper context injected
Export PDF generation In-browser jsPDF — A4 formatted document download

16 JSON fields extracted: title, authors, abstract, keywords, methodology, contributions, results, limitations, future work, dataset, evaluation metrics, research gap, theoretical framework, practical applications, related works, conclusion.

DEOZA — Research Intelligence Platform

DEOZA is a live academic search engine with four AI layers on top of every search query and paper.

Feature What it does API
Dual-source search Queries Semantic Scholar + arXiv in parallel, merges and ranks up to 30 papers /api/deoza/search
Field Intelligence Generates a 4-section domain briefing for any search query /api/deoza/insight
Paper Summarizer 7-point structured analysis of any paper (summary, methodology, gaps, future work) /api/deoza/summarize
Paper Q&A Chat Multi-turn context-grounded chatbot grounded in paper title + abstract /api/deoza/chat
Personal Library Save any paper to localStorage-persisted Zustand store Client-side only

Shared Infrastructure
History system · Zustand stores · localStorage · Unified sidebar

All three tools share a single, unified history system built on LocalStorage + Zustand. This means history is persisted across sessions and survives page refreshes — without any database.

History Architecture

Tool localStorage Key Item Fields Stored Max Items
SENTIC sentic_history id, original (input text), humanized (output text), timestamp 20
ALAMIX alamix_history id, fileName, title (extracted), authors (extracted), timestamp 20
DEOZA deoza_history id, query, resultCount, timestamp — deduplicated by query string 30

UnifiedHistorySidebar Component

One single UnifiedHistorySidebar component handles all three tools. It receives a tool prop ("sentic" | "alamix" | "deoza") and reads from the corresponding localStorage key. The sidebar:

  • Slides in from the left with a spring animation (type: "spring", damping: 28, stiffness: 300)
  • Opens/closes via useHistoryPanel() Zustand atom — toggled by the Clock icon in the NavigationRail
  • Shows tool-coloured headers: purple for SENTIC, blue for ALAMIX, emerald for DEOZA
  • Per-item delete (hover to reveal trash icon) + Clear All button at bottom
  • For SENTIC: clicking a history item calls onSenticSelect(item) to restore the humanized text to the UI
  • For DEOZA: clicking calls onDeozaSelect(query) to re-trigger that search

Zustand Stores

Store File Used By Key State
History panel lib/history-store.ts All tools + NavigationRail isOpen, toggle(), close()
DEOZA store lib/deoza-store.ts DEOZA pages + components searchQuery, searchResults, isSearching, filters, savedPapers

SENTIC and ALAMIX do not use Zustand — their state is managed locally within their page components using React useState. Only DEOZA needs cross-page state because its search results must survive navigation (search → detail → back).

API Layer — All Endpoints at a Glance
7 API routes · Serverless Next.js + 1 Python server
POST /api/humanize
SENTIC main pipeline. Strips AI-isms, chunks text, runs Groq Stage 1 (entropy rewrite), calls Python Stage 2 (marker injection). Returns humanizedText + analysis + explainability.
POST /api/alamix/extract
ALAMIX extraction. Calls Python for text extraction first, then Groq at temperature=0 in JSON mode. Returns structured 16-field AcademicResult JSON.
POST /api/deoza/search
DEOZA search. Queries Semantic Scholar API + arXiv Atom/XML in parallel. Applies filters, runs smart ranking algorithm, returns up to 30 UnifiedPaper objects.
POST /api/deoza/insight
Field Intelligence briefing. Takes a query string, returns a 4-section domain briefing from Groq. temperature=0.5, max_tokens=600.
POST /api/deoza/summarize
Paper Analyzer. Takes title + abstract, returns 7-point structured analysis from Groq. temperature=0.4. Used by AISummaryPanel on paper detail page.
POST /api/deoza/chat
Chatbot. Takes paper title, abstract, and full message history[]. Injects paper context into system prompt. temperature=0.6, max_tokens=1024.
POST :8000/inject-markers (Python)
Python FastAPI — Stage 2 of SENTIC. Injects human pragmatic markers ("Frankly", "In practice") into text. Called with 3s timeout from /api/humanize.
POST :8000/extract-text (Python)
Python FastAPI — Stage 1 of ALAMIX. Extracts raw text from PDF (PyMuPDF) or DOCX (python-docx). Returns clean normalized text string.

All Groq Calls — Parameters Summary

Endpoint Model Temp Max Tokens Response Format
SENTIC humanize llama-3.3-70b-versatile Dynamic (entropy) Default Plain text
ALAMIX extract llama-3.3-70b-versatile 0.0 Default JSON object
DEOZA insight llama-3.3-70b-versatile 0.5 600 Plain text
DEOZA summarize llama-3.3-70b-versatile 0.4 Default Plain text
DEOZA chat llama-3.3-70b-versatile 0.6 1024 Plain text

All five Groq calls share the same model. Temperature is the key differentiator — from 0.0 (deterministic extraction) up to 0.6 (natural conversation).

Landing Page Architecture
app/page.tsx · 5 sections · Full-page marketing experience

The ALTRIX landing page (app/page.tsx, 452 lines) is a fully client-side, scroll-animated marketing page that serves as the platform's front door. It has no NavigationRail — it's standalone. Five sections:

Section ID Key Elements
Hero – "ALTRIX" title in Orbitron (clamp: 5rem–9rem), animated NeuralLines SVG background, breathing ambient glows, CTA buttons, scroll chevron
Ecosystem #ecosystem 3-column tool card grid — each card links to its tool. Gradient glow borders, floating icons, scan-line animations. FadeUp on scroll using useInView
How It Works – 3-step flow: Input → Process → Output. Glass panels with animated arrow connectors between steps
About #about What is ALTRIX? paragraph + 3 value proposition stat cards
Vision – Future roadmap teaser with an animated scanning gradient line

Key Reusable Helpers defined in page.tsx

Helper Purpose
FadeUp Scroll-triggered fade+slide animation wrapper using useInView with margin: "-60px" trigger
NeuralLines SVG component rendering 7 animated Bezier paths + glowing nodes simulating a neural network background
SectionLabel Section label with horizontal lines on each side — reused in Ecosystem, About, Vision sections
No Auth, No Database

ALTRIX has no authentication layer and no server-side database. All user data (history, saved papers) lives in the browser's localStorage. This makes the platform instantly usable — no sign-up required — and fully stateless on the server side for the Next.js layer.
